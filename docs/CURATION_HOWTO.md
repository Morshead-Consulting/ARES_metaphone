# Curation report — how to read and use it

## What the report is

`curation_report.txt` is generated by `altspell_gen.py` and lists English words
that are phonetically confusable with each target military acronym. It is an
**assist to expert curation, not a finished product** — the algorithm proposes
candidates; a wargame-experienced reviewer keeps the plausible ones, discards
the noise, and adds real mis-recognitions the algorithm misses.

Since Phase 2 the matching runs on how each acronym is **spoken**, not on its
spelling. Each target block lists the spoken forms used (ISTAR is matched as
"eye star", "is tar", "eye ess tee ay are", ...) and each proposal shows which
spoken form produced it. The authoritative design rationale is in
`ARES_DN_PHON_001.md` §3, with the segmentation approach in `CHANGES_phase2.md`.

---

## How to regenerate it

```
uv run python altspell_gen.py --targets data/targets.txt --use-wordlist --max-key-dist 0 --max-per-target 15 --report --output-dir data/reports
```

The `--output-dir` flag writes the output to a timestamped file in the given
directory instead of stdout. The filename is derived from the targets file and
the current time, e.g. `targets_20260610_143022_curation_report.txt`, so runs
from different inputs are kept distinct. The path is echoed to stderr.

Omit `--output-dir` to write to stdout as before (redirect with `>`):

```
uv run python altspell_gen.py --targets data/targets.txt --use-wordlist --max-key-dist 0 --max-per-target 15 --report > data/curation_report.txt
```

Install `wordfreq` (`pip install wordfreq` / `uv sync`) to rank proposals
common-word-first and show a Zipf score — strongly recommended, as word
commonness is the main thing you are judging.

---

## Report format

```
TARGET  spoken as: "form 1" | "form 2" | ...
    <- candidate_word          key_dist=N surface_dist=N zipf=N.N  via "spoken form"
```

| Field | Meaning |
|---|---|
| `TARGET` | The military acronym from `targets.txt` |
| `spoken as:` | The k-best spoken forms of the acronym used for matching (the full letter-by-letter spell-out is always included as a fallback) |
| `candidate_word` | An English word whose phonetic key is identical or near-identical to one of the target's spoken forms |
| `key_dist` | Levenshtein distance between the two phonetic keys (0 = identical sound; 1 = one-edit near-match) |
| `surface_dist` | Levenshtein distance between the raw strings — a rough guide to how different they look |
| `zipf` | Word-frequency score, if `wordfreq` is installed (>=3.5 common, <2 obscure, 0 = not a known word) |
| `via "..."` | The spoken form that produced the match — shown when it differs from the raw target or candidate (e.g. `via "see bee are en"`) |

Proposals are sorted by phonetic distance, then by word frequency (common
first), then by surface closeness. The list is capped per target
(`--max-per-target 15`).

---

## How to curate

Go through each target block and ask: **could an ASR engine plausibly output this
word instead of the acronym in a military-comms context?**

**Keep** a candidate if:
- It sounds like the acronym when spoken aloud, AND
- It is a word an ASR language model might plausibly predict (a high `zipf` is
  a strong signal here).

**Discard** a candidate if:
- It is an obscure dictionary word unlikely to appear in ASR output (`zipf`
  near 0, or flagged "non-word" / "too obscure").
- It only matched through the full **spell-out** fallback and is not a
  realistic mis-recognition. Word-like acronyms (e.g. CASEVAC) occasionally
  attract odd spell-out matches such as "sisyphus" `via "see ay ess ee vee ay
  see"` — discard these on sight.
- It is implausible in the operational context even if phonetically close.

**Add manually** any real mis-recognitions observed in Sprint 3 recordings that
the algorithm did not surface. Segmentation now gives a useful starting set
even for letter-by-letter acronyms (CBRN -> sabrina, siberian, seaborne; IED ->
its spoken-form matches), but observed errors from real recordings remain the
ground truth and should always be added.

> **Short-acronym caution:** very short spell-outs (IED, SAM) have coarse
> phonetic keys that can attract short high-frequency words ("out", "what").
> Lean on the `zipf` ranking and your operational judgement, and consider
> `--max-surface-dist` to trim the noise.

## Optional: AI-assisted pre-annotation

A reviewer may use a general-purpose LLM (Claude, ChatGPT, etc.) to **pre-mark**
the report before human review — proposing a `[KEEP]` / `[DISCARD]` /
`[UNCERTAIN]` tag and a short rationale for each candidate. This is a labour
saver, not a decision maker: it produces a consistent first-pass draft that a
**wargame-experienced human then checks and signs off**. The judgement remains
human; the LLM only drafts.

This step is deliberately **kept outside `altspell_gen.py`**. The generator
stays deterministic, offline-capable and dependency-light — no model, no API
key, no network call — which is what makes its output auditable and the tool
itself good de-risking evidence. AI assistance is a workflow choice the
reviewer makes by hand, with full visibility of what was sent and returned.

**Provenance matters.** Record that a block was AI-pre-annotated and who
reviewed it, e.g. a header line `# AI-pre-annotated (model X, date); reviewed by
[name]`. "Machine-suggested, human-approved" is an honest and defensible
description; an unattributed annotated file that merely *reads* as expert is
not. Do not let fluent-sounding rationales substitute for the operational
judgement of someone who knows the vocabulary.

### How to do it

1. Generate the plain report as above (`--report`).
2. Paste the report into the LLM with the prompt below.
3. Save the returned annotated text alongside the original (e.g.
   `..._curation_report_annotated.txt`).
4. **Review every tag.** Correct the ones the model got wrong — especially
   `[KEEP]`s, since a false keep pollutes the context file. Mark your sign-off.
5. Carry the survivors into curation as normal.

### Suggested prompt

Copy everything between the lines. It pins down the tag set, the house style
and the rationale vocabulary so the output is uniform across every candidate.

```
You are helping pre-annotate a curation report for an automatic speech
recognition (ASR) project. Each TARGET is a military acronym. Beneath it are
candidate English words the ASR engine might output INSTEAD of the acronym,
because they sound similar. A human reviewer will check your work afterwards;
your job is a consistent FIRST-PASS draft, not the final decision.

For every candidate line, append exactly one tag and a short rationale:

  [KEEP]       a real, reasonably common English word (or legitimate variant)
               that an ASR engine could plausibly emit for the spoken acronym
  [DISCARD]    not a plausible ASR output — e.g. a non-word, a very obscure or
               technical term, a proper noun, an archaic spelling, a foreign
               word, or a "spellout artefact" (a match via the letter-by-letter
               spoken form rather than a real word)
  [UNCERTAIN]  plausible only under a condition the reviewer must confirm
               (e.g. depends on whether an abbreviation is in local use)

Rationale style — keep it terse (2-6 words), consistent, and drawn from this
vocabulary where it fits:
  "very common word, plausible ASR slip" / "common word, plausible"
  "real English word, plausible" / "legitimate variant of <word>"
  "non-word" / "too obscure" / "proper noun" / "archaic, unlikely ASR output"
  "<field> term, too obscure" (e.g. chemistry, botanical, maths)
  "spellout artefact, not an ASR output"
  "redundant if <other candidate> kept"
  "sounds too different"

Rules:
  - Judge plausibility by how the ACRONYM is spoken (shown in the "spoken as:"
    line), not by spelling. Use the zipf score as a strong commonness signal:
    zipf >= 3.5 is common, < 2 is obscure, 0 means not a known word (almost
    always [DISCARD]).
  - Tag EVERY candidate. Do not add, remove, reorder or re-rank lines.
  - Preserve the original layout exactly; only append the tag + rationale.
  - Where a whole target block is weak or empty (common for letter-by-letter
    initialisms such as CBRN, IED, ATGM), add a note:
    "# MANUAL NEEDED: add real mis-recognitions from recorded audio."
  - Do not invent candidates the report does not contain.

Output the annotated report only. Here is the report:

<paste the report here>
```

### A note on what the LLM cannot do

The model has no access to your operational vocabulary, your unit's habits, or
real recordings. It cannot know that "sim" is in your comms set, or that a given
obscure word actually does come up in a particular scenario. Those calls are
yours. Treat the AI pass as a way to clear the obvious noise quickly so your
attention goes to the genuine judgement calls — and always add real
mis-recognitions from recorded audio, which no amount of LLM annotation can
substitute for.

## Output: the CTC-WS context file

The CTC-WS context file is the end product of the whole workflow. It is what
you load into the ASR engine's context-biasing layer to tell it which
confusable words to watch for alongside each target acronym.

### Step 1 — generate the pre-curation context file

Run the same command used to produce the curation report, but omit `--report`:

```
uv run python altspell_gen.py --targets data/targets.txt --use-wordlist --max-key-dist 0 --max-per-target 15 --output-dir data/reports
```

Without `--report`, the output file uses the suffix `_ctcws.txt`, e.g.
`targets_20260610_143022_ctcws.txt`. Each line has the form:

```
TARGET_spelling1_spelling2_...
```

One line per target, with the acronym and all its proposed confusable
candidates joined by underscores.

Double Metaphone and Levenshtein distance **are** applied here — the output is
not unfiltered. With `--max-key-dist 0` only candidates whose Double Metaphone
key is **identical** to one of the target's spoken forms pass through.
`--max-per-target 15` then caps the list at 15 per target, ranked by phonetic
distance then word frequency then surface closeness (same ordering as the
report). A typical file looks like this:

```
ISTAR_astare_aster_astir_astor_astr_astur_ester_ostara_uster_wister_asteer_astore_astray_astre_auster
ORBAT_orbate_orbit_orbed_orbite_orbity_orpit_airboat_arabit_arbith_arbota_arbute_erept_erupt_jarbot_aerobate
CASEVAC_zyzzyvas_sisyphus_zizyphus
CBRN_spurn_suborn_cibarian_cyprian_cyprina_cyprine_sabrina_saprin_speron_spiran_spoorn_sprain_suberin_zebrina_zebrine
OPFOR_epiphora
SITREP_strep_satrap_strap_streep_strip_strop_estrepe_satrapy_stirp_stirrup_stripe_strype_stripy_stroup_strub
SAM_saim_same_samh_saum_seam_sem_siam_sim_sym_soam_sum_swam_asem_assam_ism
IED_yed_itd_iud_jed_wed_wied_add_aet_aid_aud_awd_awed_ead_edh_edo
ATGM_autogamy_outgame_aetheogam_outagami_outcame_outcome_euthycomi
```

The variation in candidate counts is expected: targets with a distinctive
phonetic key (OPFOR → one candidate; MEDEVAC → two) pass very few words; short
acronyms (SAM, IED) have coarse keys and attract many short common words.
`CASEVAC_zyzzyvas_sisyphus_zizyphus` is a typical spell-out artefact — those
three words matched via the letter-by-letter fallback "see ay ess ee vee ay
see", not a realistic spoken form; they should be discarded in Step 2.

### Step 2 — apply your curation decisions

The pre-curation file contains every candidate the algorithm proposed. Open it
alongside your annotated curation report and remove any candidates you marked
`[DISCARD]` from each line, leaving only the confusables you judged to be
plausible ASR outputs.

### Step 3 — load into the ASR system

Paste the edited lines into the CTC-WS context file that the ASR system reads
at inference time, replacing or extending the existing hand-authored entries.
This file is the deliverable.
